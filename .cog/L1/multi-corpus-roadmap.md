## Item #4 ‚Äî Mission Control multi-corpus refactor

### Files that need touching

- `backend/routers/graph.py` ‚Äî `/discover` route signature

  `label_clusters`, `cluster_docs`, `_qdrant_collections_for_packet`

  `selectedCorpusIds[0]`
- `frontend/src/stores/graphSessionStore.ts` ‚Äî `activeCorpusId: string`
  session keying
- `frontend/src/lib/api.ts` ‚Äî `discoverGraph()` signature

### Caches that are per-corpus today

- `graph_domain_cache` (Mongo) ‚Äî HDBSCAN clusters + LLM-labelled domains
- `graph_anchor_cache` (Mongo) ‚Äî query anchor resolutions
- `graph_metrics_cache` (Mongo) ‚Äî hub/bridge/gap metrics
- Each keyed by `{corpus_id, corpus_change_signature}`

**Cache key decision needed**: for multi-corpus discover, do we:
- (a) compute per-corpus and union at read time, OR
- (b) store a combined-key row keyed by `sorted(corpus_ids)` hash?

Option (b) is faster on cache hit but invalidates more eagerly
(any member corpus changing invalidates the combined row). Option (a)
is more storage but plays nicer with overlapping selections.

### Cypher migration pattern

Existing:
```cypher
MATCH (c:Chunk {corpus_id: $corpus_id})-[:MENTIONS]->(e:Entity)
```

Multi-corpus:
```cypher
MATCH (c:Chunk)-[:MENTIONS]->(e:Entity)
WHERE c.corpus_id IN $corpus_ids
```

`:Chunk(corpus_id)` index covers this; benchmark to confirm.

### Constraints

- **Don't break single-corpus calls** ‚Äî `corpus_id: str` is the
  current contract. Accept both:
  ```python
  if isinstance(body.corpus_id_or_ids, str):
      corpus_ids = [body.corpus_id_or_ids]
  else:
      corpus_ids = body.corpus_id_or_ids
  ```
- **Schema lens is per-corpus**. With 3 corpora, do we merge lenses,
  pick the largest, or render all three blocks? Lean toward merge:
  union of `concept_families`, intersection of `preferred_entity_types`,
  the prompt block tolerates ~600 lens tokens.
- **Sessions** are persisted with `corpus_id: str` keying. New sessions
  with multi-corpus need a new column or composite key.

### Estimated effort

Half a day for an experienced dev. Mostly mechanical once the cache
decision is made.

### Cypher template (proven on by-document)

```cypher
MATCH (c:Chunk)-[:MENTIONS]->(e:Entity)
WHERE c.corpus_id IN $corpus_ids
WITH e, count(DISTINCT c) AS mention_count
RETURN e.entity_id AS id, ...
ORDER BY mention_count DESC
LIMIT $max_nodes

MATCH (a)-[r:RELATES_TO]->(b)
WHERE any(cid IN $corpus_ids WHERE cid IN coalesce(r.corpus_ids, []))
RETURN DISTINCT a.entity_id AS source, b.entity_id AS target, ...
LIMIT $max_edges
```

### Estimated effort

~2 hours including smoke. Much simpler than #4 ‚Äî no cache layer.

## Item #6 ‚Äî Schema lens drift

### What it is

`corpora.{corpus_id}.schema_lens` is a Mongo doc generated by
`analytics.py::label_clusters()`. Injected as ~250-token guidance
block into every Ghost B prompt. Drifts over time as corpus content
shifts away from the labelled clusters. Doesn't break extraction
(prompt says "trust chunk text if conflicts"), but adds wasted prompt
tokens.

### Cost of drift

500 books √ó 500 chunks/book √ó 250 lens tokens = 62.5M wasted prompt
tokens per batch ‚âà $17 at DeepSeek input pricing.

### Refresh mechanism

`analytics.label_clusters()` is one LLM call (~$0.01) that:
1. Reads current entities from Neo4j grouped by HDBSCAN cluster
2. Asks an LLM to name each cluster + pick relevant entity types/preds
3. Writes new lens to `corpora.{cid}.schema_lens`

Existing stored entities/relations don't change. Only future Ghost B
calls see the new lens.

### Three implementation tiers

1. **CLI script** (~10 min) ‚Äî `scripts/refresh_schema_lens.py`,
   admin-run as needed
2. **API endpoint** (~1h) ‚Äî `POST /api/corpora/{corpus_id}/schema-lens/
   refresh` + UI button
3. **Auto-trigger** (~half day) ‚Äî wire into existing
   `warm_graph_cache` post-batch hook with drift detection
   (e.g. `entity_count_drift > 30%` triggers refresh)

User's preference not yet stated. Tier 1 is the obvious first move.
